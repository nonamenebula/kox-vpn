# CHANGELOG — KOX Shield

## 2026.05.14.03

### 🐛 Исправления по результатам аудита

- **`kox-cli.sh`**: исправлен критичный баг — путь к файлу загруженных категорий
  (`kox-lists-loaded.conf`) был разным у CLI и бота. CLI записывал в
  `/opt/etc/xray/kox-lists-loaded.conf`, бот читал из
  `/opt/etc/xray/lists/kox-lists-loaded.conf`. Из-за этого бот всегда видел
  «нет загруженных категорий», авто-уведомления об обновлении списков не работали,
  диффы показывали все домены как новые. Теперь оба скрипта используют один путь:
  `lists/kox-lists-loaded.conf`. При обновлении старый файл автоматически переносится.

- **`kox-cli.sh`**: в `kox upgrade` заменён `curl -sSL` на `curl -fsSL` при загрузке
  скриптов. Без флага `-f` HTTP-ошибка (404, 503) возвращала непустое тело
  (HTML-страницу), которое проходило проверку `[ -s ]` и замещало рабочий скрипт мусором.

- **`kox-bot.sh`**: экранируются спецсимволы (`"` и `\`) в названиях серверов при
  построении JSON-клавиатуры списка серверов. Кавычка в имени сервера ломала JSON и
  приводила к ошибке `sendMessage`/`editMessageText`.

- **`kox-bot.sh`**: исправлен баг со sticky-сообщением — нажатие кнопок на
  уведомлениях об обновлении (отдельные сообщения) перезаписывало ID sticky-меню,
  после чего главное меню переставало обновляться и слетало до `/start`. Теперь
  sticky обновляется только если коллбэк пришёл из текущего sticky-сообщения.

## 2026.05.14.02

### 🔁 Авто-восстановление бота через watchdog

- **`kox-watchdog.sh`** теперь следит за Telegram ботом. Если бот не работает
  (например упал после самообновления), watchdog поднимает его автоматически в
  течение минуты — без SSH и без перезагрузки роутера.
  Это решает проблему для пользователей у которых бот умер после апгрейда через
  предыдущие версии.

## 2026.05.14.01

### 🐛 Исправление: бот не запускался после самообновления

**Причина (race condition):** При обновлении через бот (`kox upgrade --force`) процесс
обновления является дочерним процессом бота. BusyBox ash откладывает обработку SIGTERM
в родительском процессе пока дочерний не завершится. Из-за этого `S90kox-bot start`
видел старый PID бота живым (lock-файл актуален), считал что бот уже работает и не
запускал новый экземпляр. После выхода `kox upgrade` родительский бот обрабатывал
отложенный SIGTERM и завершался — новый бот так и не запускался.

- **`kox upgrade`**: после SIGTERM добавлен проход с SIGKILL + принудительное удаление
  `/tmp/kox-bot.lock` перед запуском нового экземпляра бота. Это гарантирует старт
  нового бота вне зависимости от состояния lock-файла.

### 🔔 Исправление: повторные уведомления об обновлении

- **Бот больше не отправляет уведомление об одной и той же версии каждые 6 часов.**
  Теперь уведомление о доступном обновлении отправляется один раз на версию.
  При нажатии «Не напоминать сегодня/месяц» уведомление придёт ещё раз после истечения
  периода ожидания. «Отключить» полностью выключает уведомления.

## 2026.05.13.03

- Исправлена корневая причина падения Xray: `too many open files` (лимит файловых дескрипторов). Добавлен `ulimit -n 65535` в шаблон `S24xray`, в `kox restart` и в watchdog перед запуском Xray.

## 2026.05.13.02

### 🐛 Критическое исправление: интернет пропадал при падении Xray

**Причина:** Keenetic автоматически перезапускает скрипты в `netfilter.d/` при любом изменении сети (реконнект, DHCP). NAT-скрипт `99-kox-nat.sh` добавлял iptables-правила не проверяя, запущен ли Xray. Если Xray упал, watchdog снимал правила, затем Keenetic их возвращал — интернет блокировался повторно.

- **`99-kox-nat.sh`** теперь проверяет `pgrep xray` перед добавлением правил — если Xray не запущен, скрипт выходит без изменений iptables
- **`kox upgrade`** теперь автоматически обновляет `/opt/etc/kox-watchdog.sh` и патчит NAT-скрипт guard'ом при обновлении
- **`install.sh`** больше не встраивает устаревший watchdog v2 инлайн — скачивает актуальный `kox-watchdog.sh` с GitHub (с failover, Telegram-уведомлениями, auto-return)
- **`kox-watchdog.sh`** теперь удаляет оба PREROUTING правила (TCP и UDP/443) при падении Xray и повторно очищает iptables если рестарт не удался

## 2026.05.13.01

## 2026.05.02

### 🤖 Telegram Bot — крупное обновление

- **Переключение серверов из подписки** — новый экран «🌐 Серверы» показывает все сервера из подписки с именами (🇩🇪 Германия, 🇫🇮 Хельсинки и т.д.), пингом и цветовым индикатором задержки (🟢🟡🟠🔴). Одно нажатие для смены сервера

- **Исправлены бесконечные уведомления об обновлениях** — offset теперь сохраняется в `/opt/etc/xray/.kox-bot-offset` (постоянная память вместо `/tmp`), что исключает повторную обработку старых callback'ов после перезагрузки роутера
- **`/start` всегда присылает свежее меню** — если очистить чат с ботом, `/start` корректно отправляет новое сообщение
- **Исправлена ошибка повторных уведомлений о списках** — убран сброс snooze при обновлении списков (`lists_enable_notify` больше не вызывается автоматически)
- **Новая команда `/sub URL`** — установить URL подписки прямо через Telegram
- **Новая команда `/update`** — проверить и установить обновление KOX без ожидания 6 часов
- Проверка обновлений теперь использует proxy+direct fallback (работает даже когда VPN выключен)

### 🛡️ Надёжность и фолбэк

- **iptables перехватывает только порты 80 и 443** — вместо всего TCP. Нагрузка на роутер снижена, остальной трафик (игры, торренты, нестандартные порты) идёт напрямую
- **Watchdog v2** (`/opt/etc/kox-watchdog.sh`) — проверяет не только процесс Xray, но и активность порта 10808. При падении — снимает iptables (интернет работает напрямую), затем пытается перезапустить Xray
- **При отказе VPN интернет не пропадает** — watchdog снимает правила iptables, трафик идёт напрямую к провайдеру
